# 🤖 Claude 작업 가이드

이 문서는 Claude Code가 프로젝트 작업을 수행할 때 따라야 할 규칙과 절차를 정의합니다.

---

## 🎯 작업 원칙

### 1. 모델 선택
- **권장 모델**: Claude Opus 4.5
- 복잡한 작업과 깊은 사고가 필요한 경우 Opus 모델 사용
- 모델 변경은 사용자가 Claude Code 설정에서 직접 변경

### 2. 작업 시작 전 준비
작업을 시작하기 전에 **반드시** 다음을 수행:

#### ✅ 맥락 파악
- `.claude-commits/` 폴더의 최근 커밋 내역 읽기
- 최근 14일간의 작업 흐름 이해
- 프로젝트의 현재 상태와 진행 방향 파악

#### ✅ 질문하기
- 작업 내용이 불명확하면 **먼저 질문**
- 의도를 파악하지 못한 채 작업하지 않기
- 더 나은 대안이 있다면 제안
- 무조건 "OK"보다는 **능동적인 소통** 우선

### 3. 작업 중
- TodoWrite 도구로 작업 단계 관리
- 각 단계별 진행 상황 실시간 업데이트
- 예상치 못한 문제 발견 시 즉시 보고

---

## 📝 커밋 내역 관리

### 커밋 로그 작성 규칙

모든 커밋 후 **반드시** 다음 정보를 기록:

#### 1. 기본 정보
- 커밋 해시 (짧은 형식)
- 커밋 시각
- 커밋 메시지

#### 2. 상세 내역
```markdown
### [시각] 커밋 메시지

**커밋 해시**: `abc1234`

#### 🎯 작업 내용
- 무엇을 했는지 구체적으로 기술

#### 💡 의도 및 배경
- 왜 이 작업을 했는지
- 어떤 문제를 해결하려 했는지
- 의사결정 근거

#### 🔧 기술적 변경사항
- 추가된 파일/기능
- 수정된 코드
- 삭제된 항목

#### ⚠️ 주의사항
- 다른 개발자가 알아야 할 사항
- 잠재적 이슈나 제약사항
- 추가 작업이 필요한 부분

#### 🔗 관련 항목
- 연관된 이전 커밋
- 관련 이슈나 작업
```

### 파일 구조

```
.claude-commits/
├── 2025-11-27/
│   └── daily-log.md          # 하루치 모든 커밋 기록
├── 2025-11-28/
│   └── daily-log.md
├── archive/
│   └── 2025-11-summary.md    # 14일 이상 지난 내역 아카이브
└── README.md                  # 커밋 로그 시스템 설명
```

### 아카이브 규칙

**14일 이상 지난 커밋 내역**은 자동으로 아카이브:

1. 날짜별 폴더를 `archive/` 로 이동
2. 월별로 요약 문서 생성
3. 주요 변경사항만 간략히 보존
4. 상세 내역은 git history 참조

아카이브 형식:
```markdown
## 2025년 11월 요약

### 주요 작업
- 위키 시스템 디자인 개편
- 타입 에러 수정
- 인증 시스템 보완

### 주요 커밋
- `e7db489`: 클린 모던 위키 디자인으로 전면 개편
- `a95e1b1`: verifyPassword 함수 추가
- ...

*상세 내역은 git log 참조*
```

- 모바일 레이아웃 개선 필요

## 📋 다음 작업

- [ ] 모바일 UI 테스트
- [ ] 문서 업데이트

## 🔄 작업 프로세스

### 1단계: 준비
```
1. 최근 커밋 로그 읽기 (.claude-commits/)
2. 프로젝트 현재 상태 파악 (git status)
3. 작업 요구사항 명확히 하기 (질문)
```

### 2단계: 계획
```
1. TodoWrite로 작업 단계 작성
2. 예상 작업 시간 및 범위 확인
3. 필요한 도구 및 리소스 확인
```

### 3단계: 실행
```
1. 단계별 작업 수행
2. 진행 상황 실시간 업데이트
3. 테스트 및 검증
```

### 4단계: 커밋
```
1. git add & commit
2. daily-log.md에 커밋 내역 기록
3. 다음 작업 사항 정리
```

## 💬 소통 가이드

### 질문해야 할 때
- 요구사항이 모호할 때
- 여러 해결 방법이 있을 때
- 기존 코드를 크게 변경해야 할 때
- 성능/보안 트레이드오프가 있을 때
- 사용자의 의도가 불확실할 때

### 제안해야 할 때
- 더 나은 방법을 알고 있을 때
- 잠재적 문제를 발견했을 때
- 추가 작업이 필요한 부분을 발견했을 때
- 리팩토링이 필요할 때

### 보고해야 할 때
- 예상치 못한 에러 발생
- 작업 범위 초과
- 외부 의존성 문제
- 테스트 실패

## 📚 참고사항

### 관련 문서
- `CLAUDE_SETUP.md`: 대화 기록 동기화 가이드
- `.claude-history/`: 대화 기록 백업
- `.claude-commits/`: 커밋 내역 로그

### Git 작업 시
- private repository 사용 (민감한 정보 포함)
- 커밋 메시지는 명확하고 구체적으로
- 하나의 커밋은 하나의 논리적 변경사항

### 보안 주의사항
- API 키, 비밀번호 등 절대 커밋하지 않기
- .env 파일은 .gitignore에 포함
- 민감한 정보는 커밋 로그에도 기록하지 않기

## 🎯 목표

이 워크플로우의 목표는:

1. **맥락의 연속성**: 작업 간 맥락이 끊기지 않도록
2. **투명한 소통**: 무엇을, 왜, 어떻게 했는지 명확히
3. **효율적 협업**: 다른 개발자(또는 미래의 나)가 이해하기 쉽게
4. **품질 관리**: 체계적인 기록으로 품질 향상
